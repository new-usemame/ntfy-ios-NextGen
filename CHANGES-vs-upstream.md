| 2026-07-20 | Do Core Data work for "Add subscription" on the context's own queue: `Store.saveSubscription` inserted the `Subscription` on the *caller's* thread and wrapped only the save in `DispatchQueue.main.sync`, so subscribing from `SubscriptionAddView`'s background hop could silently fail to persist the topic, and would have deadlocked outright once that hop was removed. `saveSubscription`/`saveUser` now run under `context.performAndWait`, the swallowed `try?` save error is logged, and the two `checkAuth` completions hop to main before touching `@State` or the store | fix | — | TBD | TBD | TBD |
| 2026-07-21 | Notify only for messages a poll actually stored: `Store.saveNotifications` already deduplicated against the rows it had, but `SubscriptionManager.poll` handed its completion handler the raw server response, so the background `~poll` wakeup posted a banner per returned message. | 2026-07-23 | Drop a deleted notification from the notification list the moment it is deleted: `NotificationsObservable` republished its fetched results through `DispatchQueue.main.async`, so for one runloop turn the `@Published notifications` array still held a `Notification` whose Core Data row was already gone — handing SwiftUI an invalidated managed object to render while the save had already told the row's `@ObservedObject` to refresh. It also now captures `fetchedObjects` when the change fires rather than re-reading it inside the deferred closure, so a queued update can no longer publish a different fetch than the one it was scheduled for | fix | — | TBD | TBD | TBD |
| 2026-07-24 | Resolve a notification's display name (and Basic-auth user) on the Core Data context's own queue on the push path: `AppDelegate.showNotification` (inside a URLSession poll completion) and the NSE's `handleMessage` (off the extension's queue) called `getSubscription(…)?.displayName()` and `getUser(…)?.toBasicUser()`, which materialize a main-queue `viewContext` managed object and read its properties off that queue — a Core Data concurrency violation introduced alongside the #1314 custom-display-name work. New `Store.subscriptionDisplayName` does the fetch and the `displayName()` extraction inside `performAndWait` and returns a `String`; the app path switches to the already-hopping `getBasicUser`. The `ntfy` Test action now runs under `-com.apple.CoreData.ConcurrencyDebug 1` so any future off-queue Store access traps in CI | fix | — | TBD | TBD | TBD |
